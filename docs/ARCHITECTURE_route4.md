# 道4アーキテクチャ — 実LINE接続パイロット

## 全体像

```
┌─────────────┐        ①端末リンクQR          ┌──────────────────────────┐
│ 本人 iPhone  │ ───(あなたが発行)──────────▶ │  帳場 Control Plane        │
│  (PWA/Safari)│ ◀── Webプッシュ通知 ────────  │  - ユーザー/端末リンク      │
│              │                               │  - デスクのプロビジョニング  │
│              │        ②LINEログインQR        │  - 通知/フェイルセーフ配信   │
│              │ ──(LINEが発行・本人が読む)──┐  └──────────┬───────────────┘
└─────────────┘                            │             │ 1ユーザー=1台
                                            │             ▼
                                            │   ┌──────────────────────────┐
                                            └──▶│ 仮想デスク(隔離VM/コンテナ) │
                                                │  - PC版LINE(正規ログイン)   │
                                                │  - 読み取りアダプタ         │
                                                │  - 送信機能なし(構造的)     │
                                                └──────────┬───────────────┘
                                                           │ 受信イベント
                                                           ▼
                                                  既存 /api/incoming → トリアージ→下書き
```

## QRは2種類ある(ここを混同しない)

| | 発行者 | 何を繋ぐ | 難易度 | リスク |
|---|---|---|---|---|
| ①端末リンクQR | **あなた** | iPhone ↔ 帳場アカウント | 易 | 低 |
| ②LINEログインQR | **LINE** | 仮想デスクのPC版LINE ↔ 本人のLINE | — | 規約グレー |

本人の体験上は「QRを読む」だが、②は本人が自分のスマホでLINE純正の
「PCログインを許可しますか」を承認する操作。これは代行不可・自動化不可。
**②を隠したり自動化しようとした瞬間に規約違反の度合いが跳ね上がる**ため、
むしろ「PCでLINEを開くのと同じことをしています」と明示して同意を取る設計にする。

## オンボーディング手順(本人視点=3ステップ)

1. 招待リンクを開く → ①端末リンクQR or ワンタップで PWA をホーム画面追加
2. 画面の案内に沿って、表示された **②LINEログインQR** を LINE アプリで読む
   → 「PCでログイン」を承認、本人確認番号を入力
3. 完了。以降デスクが受信を見張り、通知が来る

## デスクのライフサイクル(Control Plane が管理)

```
requested → provisioning → awaiting_line_login → active
                                    │                │
                                    │                ├─ session_lost → reauth_required(→本人にQR再提示)
                                    │                └─ idle_timeout → hibernated(コスト削減)
                                    └─ failed → (再試行 or 手動対応)
```

- `awaiting_line_login`: 仮想デスクを起動し、PC版LINEのQRを取得して本人へ中継
- `active`: 読み取りアダプタ稼働。受信を Control Plane 経由で /api/incoming へ
- `session_lost`: **フェイルセーフ発火**。本人へ「LINE通知を一時的に戻して」+再ログインQR
- `hibernated`: 無操作時にVMを休止しコスト削減、受信気配で復帰

## セキュリティ要件(この事業固有・妥協不可)

- 1ユーザー=1隔離環境(VM/専用コンテナ)。相互アクセス不可
- トーク本文・認証状態は保存時暗号化。Control Plane は平文を常時保持しない
- 監査ログ(誰の環境にいつアクセスしたか)を全操作に付与
- 送信機能をコードベースから物理的に排除(watcher契約に送信メソッドを定義しない)
- データ全消去API(退会=環境破棄+バックアップ消去)

## パイロットで最初に測る3数値(READMEの再掲・最重要)

1. LINEログインセッションの**維持日数**(何日で②の再ログインが要るか)
2. **切断頻度**とトリガー(本体操作・時間・不明)
3. 受信→通知到達の**遅延**(数秒で安定するか。案A運用の可否を決める)
