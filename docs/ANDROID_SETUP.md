# Android通知方式 セットアップ＆ロードマップ（本命構成）

あなたのiPhoneはメインのまま。買ったAndroidを「同じLINEアカウントのサブ端末」にして、Androidに来る着信通知を帳場に送る、という構成です。

まず、いちばん大事な安全設計から。

---

## ⚠️ 0. 安全設計（先に読む）

- これはLINE規約グレーで、**アカウント凍結リスクは残ります**。凍結されて困るのは、客が全部入ったあなたの本アカウント。
- だから鉄則：**まず捨てて良いサブのLINEアカウントで、フェーズ0〜1まで通す**。動くと確認できてから、本アカウントに切り替えるか判断する。
- 帳場は送信機能を持ちません。読み取り→取り込みだけ。やめたくなったら受け付けを停止し、サーバーを止めれば終わりです。

---

## ロードマップ（全体像）

- **フェーズ0**：Androidをサブ端末にして、**通知が本当に届くか**だけを確認（コード不要・最重要）
- **フェーズ1**：転送アプリでLINE通知を帳場に送り、仕分け→下書き→iPhone通知まで通す（← 今ここのコードは実装済み）
- **フェーズ2**：相手名・本文の取り出し精度を実データで調整、重複除去の確認
- **フェーズ3**：クラウド常設で、iPhoneが外からでも使えるように
- **フェーズ4**：専用Androidアプリ化（売り物の形）

---

## フェーズ0：通知が届くかを確かめる（最重要・コード不要）

ここが通らなければ計画は成立しません。まずこれだけを潰します。

1. Androidの初期設定を済ませ、Google Playにログイン、Wi-Fiに接続、電源に挿す
2. Playストアから **LINE** をインストール
3. LINEを**同じアカウントのサブ端末として**ログインする
   - メイン(iPhone)のLINE → 設定 → アカウント → **「ログイン許可」をオン**
   - Android側のLINEを開き、メールアドレス／パスワード（またはQR）でログイン
   - iPhone側に「新しい端末でログイン」の確認が出たら承認
   - ※捨て垢で試すなら、その捨て垢でこの手順を行う
4. **テスト**：iPhoneをオンラインのまま、別の人（または別アカウント）から1通送る
5. **Androidに着信通知が出るか見る** → 出れば土台は成立。出なければ、ここで方式を見直す

> メインとサブ、どちらに通知が出るかはLINEの挙動次第（要検証）。両方に出る／サブにも出る、なら成功。ここが今回の山場です。

---

## フェーズ0.5：Androidの通知を「消えないよう」設定する

Androidは省電力でバックグラウンドを止めがち。通知読み取りを常駐させるため、先に緩めます（機種で名称が違います）。

- 設定 → バッテリー → LINEと（後で入れる）転送アプリを **「最適化しない／制限なし」**
- 設定 → アプリ → 各アプリ → **バックグラウンド動作を許可**
- （OPPO/ColorOSやSamsung/One UIは特に：**自動起動を許可／スリープ状態のアプリから除外／最近使ったアプリで固定**）
- 通知は「表示」かつ**プレビュー（本文）を表示**にしておく（本文が隠れると読めない）

---

## フェーズ1：LINE通知を帳場に送る（コードは実装済み）

Android側は、まずはノーコードの転送アプリで試します（専用アプリは後で私が作れます）。

### 1-1. 帳場を「Android通知モード」で起動（Mac側）

```
cd ~/Downloads/chouba
CHOUBA_WATCHER=android python3 -m uvicorn app.main:app --port 8000
```

（`.env` に `CHOUBA_WATCHER=android` と書いておけば毎回不要。）
帳場を開く → 設定タブ → デスク → 「受け付けを準備する」→「受け付けを開始する」。

### 1-2. Androidから届く先(URL)を用意

同じWi-FiならMacのローカルIP、外からや確実に繋ぐならngrok（HTTPS）。ngrokの場合は別ターミナルで：

```
~/Downloads/ngrok http 8000
```

出てきた `https://xxxx.ngrok-free.app` が土台のURL。送り先は：

```
（土台URL）/api/android/notify
```

### 1-3. 転送アプリの設定（例：MacroDroid・ノーコード）

MacroDroid（Playストア・無料枠あり）でマクロを1つ作ります（要検証：アプリ名や項目名は版で異なる）。

- **トリガー**：通知受信（アプリ＝LINE）
- **アクション**：HTTPリクエスト（POST）
  - URL：上の `…/api/android/notify`（末尾に余計な文字を付けない）
  - コンテントタイプ：`application/json`（無ければ省略可）
  - コンテントボディ（本文）：下のJSON。ただし相手名・本文の部分は、**MacroDroidの「…」ボタン（マジックテキスト）から選んで挿入する**。手打ちの角カッコ `[...]` では効かない（実機で確認済み）。
    - 「通知のタイトル」を選ぶと `{not_title}` が入る（＝相手名）
    - 「通知のテキスト」を選ぶと `{notification}` が入る（＝本文）
    ```json
    {"token":"","package":"jp.naver.line.android","title":"{not_title}","text":"{notification}"}
    ```
    ※MacroDroidのマジックテキストは**波カッコ `{}`** を使う。`{not_title}` `{notification}` が実機で動作確認済みのトークン。
- 「通知へのアクセス」許可をMacroDroidに与える

> 認証の合言葉を使うなら、帳場を `CHOUBA_INGEST_TOKEN=あなたの合言葉` で起動し、上のJSONの `token` に同じ値を入れる。第三者が勝手にPOSTするのを防げます。

### 1-4. 通す

サブアカウントに1通送る → 帳場の受信箱に出れば、Android→帳場→iPhoneの流れが実際に回ったことになります。裏側コンソール（黒帯）に**生の通知（title/text）が出る**ので、取り違えがあればその表示を私に見せてください。相手名・本文の取り出しを実データに合わせて調整します（AXツリーの時と同じやり方）。

## フェーズ2で調整する既知の弱点

- グループトークの「送信者: 本文」形式の分離（実装済みだが実データで要確認）
- 長文のプレビュー切れ（通知は末尾が省略されることがある）
- スタンプ・写真の通知（「スタンプを送信しました」等の文言をどう扱うか）
- 同一通知の再掲（重複除去は実装済み。取りこぼし/重複が出たら調整）

---

## 動作の要点（送信は必ず本人）

Android＝取得、帳場サーバー＝仕分け・下書き、iPhone＝選ぶ・直す・**自分のLINEで貼って送る**。送信機能はどこにもありません。
